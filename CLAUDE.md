# 云程智绘图（cloud-trip-agent）项目说明

## 1. 项目定位

本项目是一个面向国内旅游场景的智能行程规划与信息查询系统。

项目目标不是简单生成旅游文案，而是形成以下闭环：

用户填写需求 → Agent 生成计划 → 调用地图、天气、知识库或价格查询工具 → 返回结构化方案 → 用户保存或调整方案。

当前代码仓库是项目事实的最高优先级。README、交接说明和历史描述都可能滞后，必须结合实际代码验证。

## 3. 当前技术栈

以后续实际代码检查结果为准，历史设计主要包括：

* 后端：FastAPI
* 数据库：SQLite
* 缓存：Redis，可配置关闭
* 前端：Vue 3、Vite
* 部署：Docker Compose、Nginx
* Agent：LLM Planner、工具调用
* 检索：RAG、Embedding、Rerank
* 地图服务：高德地图相关接口
* 行程结果：结构化展示、保存和修改

重点检查：

* `backend/config.py`
* `backend/app/agents/`
* `backend/app/tools/`
* `backend/app/api/routers/`
* `frontend/src/services/api.ts`
* `docker-compose.yaml`
* 后端、前端 `.env.example`
* README 和测试目录

实际目录可能已经调整，应通过搜索确认，不要假设路径必然存在。

## 4. 产品范围

当前主要范围：

* 仅考虑国内旅游。
* 支持航班、高铁、酒店、景点、餐饮等信息。
* 暂不考虑自驾规划。
* 现阶段主要围绕携程场景，不默认接入 12306。
* 不进行自动下单、支付、退票或修改订单。
* 查询流程最多到达订单确认页之前。
* 涉及第三方平台登录时，应由用户手动登录。
* 不得读取、记录或输出用户密码、Cookie、Token、API Key。
* 不得虚构真实机票、酒店或门票价格。
* Mock 数据、缓存数据和真实查询结果必须明确区分。

## 5. 浏览器自动化状态

历史版本最初没有集成 browser-use。

目前正在评估使用 Playwright 完成携程等网页中的真实信息查询，但不能假设 Playwright 已经接入。开始相关工作前必须检查当前分支和代码。

浏览器自动化必须遵守：

* 不自动填写账号密码。
* 不绕过验证码、风控或访问限制。
* 不自动下单或支付。
* 浏览器配置、缓存和登录态不能提交到 Git。
* 优先采用独立浏览器上下文或隔离的用户数据目录。
* 页面选择器失效时应清晰报错，不能静默返回伪造结果。
* 页面查询失败时，可以降级为“暂无实时数据”，不能生成假价格。
* 不要在用户未明确要求时擅自引入 browser-use。

## 7. 修改原则

执行任务时遵循：

1. 先阅读相关代码，确认当前实现。
2. 区分“已经实现”“部分实现”“仅有接口”“Mock 数据”和“完全缺失”。
3. 先定位根因，再修改代码。
4. 优先小范围修改，不进行与任务无关的大规模重构。
5. 保持现有架构和编码风格。
6. 修改配置时同步检查 `.env.example` 和 README。
7. 新增依赖前说明必要性，避免重复依赖。
8. 不输出或提交真实 API Key、密码、Cookie、Token。
9. 不因为历史说明声称某功能已经完成，必须用代码和测试验证。
10. 发现需求与现有实现矛盾时，明确指出，不要强行附和。

## 8. 测试与验收

每次修改完成后，应尽可能执行相关检查：

* Python 语法和导入检查
* 后端单元测试或接口测试
* 前端构建检查
* 配置文件解析检查
* Docker Compose 配置检查
* 关键接口的最小链路测试

不能运行的测试必须说明：

* 未运行的具体项目
* 无法运行的原因
* 用户应执行的命令
* 尚未验证的风险

不得仅凭代码看起来正确就声称功能已经完成。
